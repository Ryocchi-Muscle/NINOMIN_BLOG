# 技術選定の根拠（個人ブログ運用）

## 前提 / 目的

- Next.js の静的ブログを本番運用しながら、SRE / DevOps に必要な基礎（変更管理・可観測性・障害対応・IaC・セキュリティ・コスト意識）を実践で身につける
- 机上の理解ではなく、実サービスに対して「安全に変更できる」「異常を検知できる」「復旧手順がある」「再現可能に管理できる」状態を作る
- 学習内容はドキュメント（設計判断・運用手順・指標）とコード（設定/IaC）に残し、継続的に改善する

## なぜAWSか（GCP/Azureとの比較）

- 国内の利用事例や運用知見が豊富で、学習中に詰まったときの参照情報が多い
- サービス選択肢が広く、可観測性・権限設計・コスト管理など SRE/DevOps の論点を一通り試しやすい
- GCP/Azureも優れているが、今回の題材（静的Web運用）ではAWSで十分であり、まずは一つのクラウドで運用の型を作る方針とした

## なぜAmplify Hostingか

- GitHub連携により、変更が Push → ビルド → デプロイ まで自動化され、変更管理（CI/CD）の入口を短時間で用意できる
- 運用負荷が低く、監視・Runbook・IaCなどの改善に時間を割ける
- カスタムドメイン対応があり、本番運用として成立させやすい

## なぜRoute53か

- DNS管理をAWS内に統一でき、Amplifyのカスタムドメイン設定と整合が取りやすい
- レコード管理をIaC化しやすく、変更履歴をレビュー可能にできる

## 将来、別案に切り替える条件

- SSR/動的機能やAPIが必要になり、バックエンド前提の構成が必要になった
- トラフィック増でキャッシュ戦略やコスト最適化が必須になった
- WAF/Bot対策などセキュリティ要件が強くなった
- IaCの範囲を広げ、環境全体を統一管理したくなった

## この構成で優先的に取り組むこと

1. 可観測性：死活監視・エラーレート・通知
2. 障害対応：Runbook整備とロールバック手順
3. 変更管理：CI/CDの見える化とデプロイ失敗時の対応
4. IaC：Route53レコードからTerraform化し、PRレビュー運用にする
